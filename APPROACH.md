# APPROACH.md — Design Decisions & Trade-offs

## 1. PDF Parsing Strategy

**Decision**: Extract spans from PyMuPDF `get_text("dict")`, detect tables by clustering x/y positions of text spans, then filter table cells out of paragraph blocks.

**Why not use PyMuPDF's built-in table detection?**
PyMuPDF's `page.find_tables()` is available but produces inconsistent results on PDFs generated by fpdf2, where table cells are individual text blocks without explicit table markup. Clustering span positions gives finer-grained control over multi-line cell merging (e.g., E2 error row split across two y-coordinates).

**Trade-off**: The span-clustering approach is fragile against PDFs with unusual layouts (overlapping text, rotated text). For production, we'd need a more robust table extraction library like `pdfplumber` or Camelot.

## 2. Hierarchical Tree Construction

**Decision**: Parse section numbers (e.g., "2.1.1.1") from heading text to determine tree depth. Use a stack-based algorithm to maintain parent-child relationships.

**Key irregularity handled**: Section 3.4 "Auto Shutoff" appears BEFORE 3.3 "Result Display" in the PDF. The parser preserves document order (not section number order) because physical order determines the user's reading experience, which is what QA test cases should verify.

**Another irregularity**: "Error Codes" appears as both `4.2 Error Codes` and `7.1 Error Codes`. The parser disambiguates via section numbers, not titles alone.

## 3. Numbered List Detection

**Decision**: Regex-match `^\d+\.\s+` on paragraph blocks, collecting consecutive matches into a single `numbered_list` node. Stop merging when encountering a heading or another numbered item.

**Problem solved**: The BP classification items (1. Normal, 2. Elevated, ..., 5. Hypertensive Crisis) are in body text blocks that the paragraph-merging logic would combine into a single blob. The fix detects numbered items during the merge phase and prevents cross-item merging.

## 4. Content Hashing for Staleness

**Decision**: SHA-256 truncated to 16 hex characters, applied to whitespace-normalized text (`" ".join(text.split())`).

**Why not full SHA-256?** The 16-char truncation is sufficient for collision-free comparison within this dataset. Full hashes add no value for staleness detection.

**Why whitespace normalization?** fpdf2 generates PDFs with inconsistent whitespace (double spaces, leading/trailing spaces). Without normalization, identical content would produce different hashes across re-ingestions.

## 5. Version Diff — Title-Based Matching

**Decision**: Match nodes between v1 and v2 by title (or body text prefix for nodes without titles). Compare content hashes to detect modifications.

**Trade-off**: Title-based matching breaks when titles change between versions. For a more robust solution, we'd use fuzzy matching or structural similarity (e.g., Levenshtein distance on body text). The current approach is sufficient for the CT-200 assignment where section titles remain stable.

## 6. Staleness Detection Model

**Decision**: Lazy staleness checking — hash is compared at retrieval time, not at ingest time.

**Why lazy?** When v2 is ingested, v1 nodes aren't modified. The node IDs in selections still point to v1 nodes. Staleness only becomes relevant when retrieving test cases and asking "are these still current?"

**Implementation**: `check_staleness()` compares `Generation.content_hash_at_generation` against `DocumentNode.content_hash` for the referenced node. If different, `is_stale=True` is persisted.

## 7. LLM Integration

**Decision**: Use Groq API (Llama 3.1 8B) with structured prompt design, JSON-only output parsing, and validation with retry.

**Structured prompt approach**: System prompt specifies exact JSON schema. User prompt includes section context. Temperature=0.3 to reduce hallucination. Post-processing validates required fields and filters invalid test cases.

**Trade-off**: Single-shot generation (no retry loop) to keep latency low. For production, we'd add retry with exponential backoff and fallback to a different model.

## 8. Technology Choices

- **FastAPI**: Async-native, auto-generates OpenAPI docs, type-safe with Pydantic
- **SQLAlchemy async + SQLite**: Lightweight, no external DB server needed, async-compatible
- **PyMuPDF (fitz)**: Fast PDF parsing with span-level text extraction
- **fpdf2**: For generating test PDFs with proper Unicode/Arial support (reportlab has Python 3.8 compatibility issues)
- **httpx**: Async HTTP client for LLM API calls
